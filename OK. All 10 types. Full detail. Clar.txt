## How concern_score is calculated (your existing formula, restating for clarity)

For ONE concern in ONE product, you look at every ingredient that addresses that concern:

```
Per ingredient:
  effective_value = tier_value × concentration_multiplier

  tier_value:
    Tier 1 = 1.0
    Tier 2 = 0.7
    Tier 3 = 0.4

  concentration_multiplier:
    ≥100% of required = 1.0
    70-99% = 0.8
    50-69% = 0.6
    30-49% = 0.4
    <30% = 0.2
    No data = 0.4

concern_score = min(sum of all effective_values / 1.0, 1.0)
```

### 3A. Efficacy Pre-Computation

**How concern_score is computed (existing formula):**

For each ingredient in the product that addresses the concern:

```
effective_value = tier_value × concentration_multiplier

  ≥100% of required = 1.0
  70-99% = 0.8
  50-69% = 0.6
  30-49% = 0.4
  <30% = 0.2
  No data = 0.4

concern_score = min(sum of all effective_values / 1.0, 1.0)
```

**Synergy (per product × per concern, ALL 16 concerns):**

- Max ~0.16 per concern
- Based on strong/moderate/weak synergy between ingredient pairs for that concern
- Computed by LLM which takes all active ingredients and finds combination mechanisms for the concern
- Pre-computed for ALL 16 concerns, not just user's concerns

**Bonus score (per product, NOT per concern):**

For each bonus concern (concerns the product's ingredients are best at, regardless of any user):

```
bonus_concern_score = min(sum of effective_values / 2.5, 1.0)
weighted = bonus_concern_score × 0.5
bonus_score = min(sum of all weighted / 2.5, 0.5)
```

Bonus is based on whatever concerns the product's ingredients are best at. NOT tied to user profile. Pre-computed once per product.

Store in `product_scores` table:

```
product_id
bonus_score (0-0.5)
```

### 3B. Safety Pre-Computation (Product-Side Flags Only)

Store in `product_safety_flags` table:

```
product_id

-- STOP flags (product facts)
pregnancy_safe (boolean)
lactation_safe (boolean)
eczema_verdict (text: none/worsens/triggers)
rosacea_verdict (text: none/triggers/worsens)
psoriasis_impact (text: none/worsens)
fungal_acne_risk (text: none/problematic)
fungal_problematic (boolean)
fungal_malassezia (boolean)
worst_comedogenicity_rating (integer 0-5)
is_carcinogenic (boolean)
is_mutagenic (boolean)
dermal_severity (text: none/mild/moderate/severe/critical)
has_banned_ingredient (boolean)

-- Deduction flags (product facts)
has_fragrance (boolean)
has_eu26_allergen (boolean)
has_preservative_allergen (boolean)
has_sun_sensitivity (boolean)
has_high_irritation (boolean)
```

**Fully pre-computable parameters (same for ALL users, can be tagged once):**

## 9. Scoring

**Scoring is a separate Code Node. It computes a number for every product that survived filtering. No sorting. No selection. Just attaches scores.**

### 9A. The Recommendation Score Formula

```
recommendation_score = relevance + safety_score + formulation_score
```

### 9B. Relevance — Concern Path

Used when: user stated a concern, OR user stated concern + ingredient, OR no concern/ingredient stated (use profile concerns as fallback).

```
n = number of concerns being evaluated
per_concern_total = sum of concern_scores for those n concerns
                    (each concern_score is max 1.0, pre-computed, just looked up)

concern_ratio = per_concern_total / n
scaled_concern = concern_ratio × 3.0       → max 3.0

synergy_total = sum of synergy_scores for those n concerns
                (each pre-computed, just looked up)

relevance = scaled_concern                  (0-3.0)
          + bonus_score                     (0-0.5, pre-computed, looked up)
          + synergy_total                   (0-~0.5)
                                            → max ~4.0
```

**Why scaling:** A product scoring 0.9/1.0 for one concern has concern_ratio = 0.9, scaled = 2.7. A product scoring 0.9 + 0.8 for two concerns has concern_ratio = 0.85, scaled = 2.55. The single-concern product scores slightly higher on scaled_concern because it's a better ratio. BUT the two-concern product gets more synergy (synergy for both concerns), and if it covers both well, total relevance can be higher. Products covering multiple concerns well are rewarded through the combination of scaled_concern + more synergy.

**Which concerns go into the calculation:**

| Scenario | Concerns used |
|----------|---------------|
| User stated 1 concern | That 1 concern |
| User stated 2 concerns | Those 2 concerns |
| User stated 3 concerns | Those 3 concerns |
| User stated concern + ingredient | Stated concern (ingredient handled by filter) |
| No concern, no ingredient (just product type) | All profile concerns (1-3) |

### 9C. Relevance — Ingredient Path

Used when: user stated an ingredient but NO concern.

**This path requires ingredient synergy to be computed FIRST (by LLM 2) before scoring can happen.**

```
concentration_multiplier of requested ingredient (pre-computed per product)
  ≥100% of required = 1.0
  70-99% = 0.8
  50-69% = 0.6
  30-49% = 0.4
  <30% = 0.2

scaled_concentration = concentration_multiplier × 3.0    → max 3.0

relevance = scaled_concentration                          (0-3.0)
          + bonus_score                                   (0-0.5, pre-computed)
          + ingredient_synergy                            (0-0.5, from LLM 2)
                                                          → max ~4.0
```

**Ingredient synergy (LLM 2, query-time):**

Same LLM that computes concern synergy in your existing pipeline, but different mode:

```
Concern synergy (existing, pre-computed):
  Input: active ingredients + concern
  Output: synergy score for that concern

Ingredient synergy (new, query-time):
  Input: target ingredient + other active ingredients in product
  Output: overall synergy of target ingredient with rest of formulation
  Same scale: strong/moderate/weak → same numeric values → max 0.5
```

### 9D. Safety Score — Query-Time Computation

Uses pre-computed product flags + merged user profile. Same parameters serve as BOTH filter (harsh end) and score (mild end).

**Starting score: 4.0**

Deductions applied to surviving products only:

| Parameter | Condition | Deduction |
|-----------|-----------|-----------|
| P1 Pregnancy (partial) | Has H361 GHS + user pregnant | -0.8 |
| P7 Comedogenicity | Rating 2-3 + user acne_prone/oily | -0.5 |
| P7 Comedogenicity | Rating 1 + user acne_prone/oily (only if no 2+ found) | -0.2 |
| P10 Fragrance | Has fragrance + user IS fragrance_sensitive | -0.3 |
| P10 Fragrance | Has fragrance + user NOT fragrance_sensitive | -0.2 |
| P11 Preservative | Has preservative allergen (all users) | -0.2 |
| P12 Sun sensitivity | Sun-sensitizing + user HAS pigmentation concern | -0.3 |
| P12 Sun sensitivity | Sun-sensitizing + NO pigmentation concern | 0 (info only) |
| P13 Irritation | Sensitive user + high irritation | -0.2 |
| P13 Irritation | Sensitive user + only medium irritation | -0.1 |
| P14 Skin type mismatch | `avoid_for_skin_types` matches user skin_type | -0.3 |
| P15 Interactions | `worst_case = avoid` | -0.5 |
| P15 Interactions | `worst_case = caution-high/medium` | -0.2 |

**Synergy offsets:**

```
Irritation synergy offset:
  Soothing ingredient pairs compensate for irritating ones
  → offsets P13 irritation deduction
  Rule: both ingredients in pair must be NON-irritating

Interaction synergy offset:
  Synergistic pairs reduce interaction risk
  → offsets P15 interaction deduction
```

**Safety score calculation:**

```
safety_score = 4.0
  + pregnancy_deduction (0 or -0.8)
  + comedogenicity_deduction (0, -0.2, or -0.5)
  + fragrance_deduction (0, -0.2, or -0.3)
  + preservative_deduction (0 or -0.2)
  + sun_deduction (0 or -0.3)
  + (irritation_deduction + irritation_synergy_offset)
  + mismatch_deduction (0 or -0.3)
  + (interaction_deduction + interaction_synergy_offset)

Minimum = 0.0, Maximum = 4.0
```

**Score bands:**
```
4.0 = SAFE
3.0 - 3.9 = LOW RISK
2.0 - 2.9 = MODERATE RISK
1.0 - 1.9 = HIGH RISK
< 1.0 = UNSAFE
```

### 9E. Formulation Score

Look up pre-computed `formulation_score` (0-2.0). No computation needed.

### 9F. Final Score

```
recommendation_score = relevance + safety_score + formulation_score

Max possible: ~4.0 + 4.0 + 2.0 = ~10.0
```

**Scoring node output:** Every product with its recommendation_score + all sub-scores (relevance, safety_score, formulation_score, concern breakdown, safety breakdown).

## 10. Ranking

**Ranking is a separate Code Node. Takes scored products. Sorts and selects. No computation.**

### 10A. Sort

Sort all products by `recommendation_score` descending.

### 10B. Brand Diversity

Walk through the sorted list. Keep a counter per brand. If a brand already has 2 products in the result, skip the 3rd and take the next product from a different brand.

```javascript
const ranked = products.sort((a, b) => b.recommendation_score - a.recommendation_score);
const result = [];
const brandCount = {};

for (const product of ranked) {
  const brand = product.brand;
  brandCount[brand] = (brandCount[brand] || 0) + 1;
  
  if (brandCount[brand] <= 2) {
    result.push(product);
  }
  
  if (result.length >= 5) break;
}

return result;
```

### 10C. Output

Top 3-5 products with all their scores and data, ready for Response Generator.

## 11. Response Generation (LLM 3)

For each recommended product, the LLM generates:

**A. Match Score:**
The recommendation_score in a readable format (e.g., 8.2/10). Builds user trust.

**B. Ingredient Story:**
Explains HOW key ingredients solve the user's concern or deliver the requested ingredient. Uses pre-computed data (tier, concentration, synergy) to explain mechanisms.

Example: "This has 5% Niacinamide which inhibits melanin transfer to surface skin cells, directly targeting your pigmentation. It also has Tranexamic Acid at 3% which works synergistically — Niacinamide reduces new pigment production while Tranexamic Acid breaks down existing pigment clusters."

**C. Why This Product:**
Explains WHY this product ranked where it did. References safety, formulation, concentration adequacy.

Example: "Ranked #1 because it has the highest concentration of proven actives for pigmentation among products safe for pregnancy. The formulation is well-preserved with good stability."

**D. Safety Note:**
If any safety deductions were applied, mention them.

Example: "Note: Contains sun-sensitizing ingredients. Use sunscreen during the day."

## 13. Edge Cases

**13A. Ingredient not found in DB:**
User asks for "bakuchiol serum" but no product has bakuchiol.
→ Filter returns 0 products.
→ "I don't have any bakuchiol serums in our database right now."

**13B. Too many safety filters eliminate everything:**
User is pregnant + lactating + rosacea + eczema + sensitive + fragrance sensitive + acne prone.
→ Very few or 0 products survive.
→ "With your safety requirements, I couldn't find products that meet all criteria. Would you like me to show the closest matches with a note on which safety concern they don't fully meet?"

**13C. Concern + Ingredient stated together:**
"niacinamide serum for pigmentation"
→ Filter: product_type=serum + contains niacinamide + concern_score_pigmentation > 0
→ Rank: CONCERN PATH (pigmentation efficacy). Ingredient handled by filter, not ranking.

**13D. Multiple concerns stated:**
"serum for acne and pigmentation"
→ Concern path with n=2. Both concern_scores summed. Scaling divides by 2.
→ Products addressing both well rank highest.

**13E. Very few results after filtering:**
Only 1-2 products match.
→ Return what matches. Don't pad with non-matching products.
→ "I found 2 products matching your criteria."

**13F. Budget too low:**
No products under stated budget.
→ "I couldn't find products under ₹300 for this. The closest options start at ₹450. Would you like to see those?"

**13G. Season mentioned:**
No seasonal data stored per product. Season does NOT affect filtering or ranking.
LLM Response Generator uses ingredient knowledge to explain seasonal suitability.
→ "This moisturizer has heavier emollients like shea butter which provide extra barrier protection in winter."

## Summary: What Is Pre-Computed vs Query-Time

### Pre-Computed (One-Time Batch Job)

| Data | Scope | Storage |
|------|-------|---------|
| concern_score | Per product × ALL 16 concerns | `product_concern_scores` |
| synergy_score | Per product × ALL 16 concerns | `product_concern_scores` |
| bonus_score | Per product | `product_scores` |
| formulation_score | Per product | `product_scores` |
| concentration_multiplier | Per product × per ingredient | `product_ingredients` |
| ALL safety flags | Per product | `product_safety_flags` |

### Computed at Query Time

| What | How | Why can't pre-compute |
|------|-----|----------------------|
| Which concerns to look up | From user query | Depends on what user asked |
| scaled_concern | Code Node: (sum/n) × 3.0 | Depends on which/how many concerns |
| safety_score | Code Node: 4.0 - deductions | Depends on merged user profile |
| scaled_concentration | Code Node: multiplier × 3.0 | Depends on which ingredient user asked |
| ingredient_synergy | LLM call | Can't pre-compute for every possible ingredient |
| recommendation_score | Code Node: addition | Sum of above |

### LLM Calls at Query Time

| Call | When | Purpose |
|------|------|---------|
| LLM 1: Intent Extractor | Every query | Understand message, extract fields |
| LLM 2: Ingredient Synergy | Only ingredient queries with no concern | Synergy of target ingredient with product |
| LLM 3: Response Generator | Every query | Score explanation, ingredient story, justification |

**Concern-based queries: 2 LLM calls (extractor + response).**

## Scenario 1 — User Refers to Recommended Product

User got 3 recommendations. Says:

```
"I've tried product 2 before, it didn't work. Show me others"
"Remove the Minimalist one, show me alternatives"
"Not the first one, I already have it"
```

**How it works:**

The Memory Node has the last conversation including the 3 recommended products with their product_ids. The intent extractor LLM reads the conversation history and maps:

```
"product 2" → product_id of the 2nd recommended product
"the Minimalist one" → product_id of the Minimalist product from recommendations
"the first one" → product_id of the 1st recommended product
```

LLM outputs:

```json
{
  "action": "refine",
  "exclude_product_ids": [42]
}
```

The system then either:

**Option A — Re-filter from already scored list (fast):**

The scoring node already scored all surviving products, not just the top 5. The ranking node picked top 5. Now just re-run ranking with product 42 excluded. Next best product moves up. No re-computation needed.

**Option B — Re-run full query (if scored list not stored):**

Re-run the same query with product_id 42 added to exclusion filter.

**Option A is better.** It means the scoring node output (all scored products, not just top 5) should be stored in the Memory Node or a temporary variable for the session. Then refinement queries just re-rank from the stored list without hitting Supabase again.

---

## Scenario 2 — User Mentions a Product Not From Recommendations

```
"I already use Minimalist 10% Niacinamide, suggest something different"
"I've been using Deconstruct serum, want to try something else"
```

**How it works:**

The intent extractor LLM detects a product name. But it needs to map the product name to a product_id in your database.

Two options:

**Option A — LLM extracts product name, Supabase fuzzy search:**

```
LLM output: { "exclude_product_name": "Minimalist 10% Niacinamide Serum" }

Supabase query:
SELECT product_id FROM products 
WHERE product_name ILIKE '%Minimalist%Niacinamide%'
LIMIT 1
```

If found → add to exclude_product_ids. If not found → ignore, proceed without exclusion, mention "I couldn't identify the exact product you mentioned."

**Option B — LLM extracts product name, exact match:**

Less reliable. Product names have variations ("Minimalist 10% Niacinamide Face Serum" vs "Minimalist Niacinamide 10% Serum"). Fuzzy search is better.

## The Implementation

**In the n8n workflow:**

```
After Intent Extractor detects exclude_product:
    ↓
IF exclude by number ("product 2"):
    → Look up product_id from Memory Node (stored recommendations)
    → Add to exclude_product_ids
    → Re-rank from stored scored list (no Supabase call)
    
IF exclude by name ("the Minimalist one"):
    → Check Memory Node first (is it one of the recommendations?)
    → If yes → same as above
    → If no → Supabase fuzzy search by name → get product_id
    → Add to exclude_product_ids
    → Re-run full query with exclusion filter
```

**The filter itself is simple:**

```sql
WHERE product_id NOT IN (42, 87, 103)
```

## What Needs To Be Stored Per Session

For exclusions and refinements to work smoothly, the session needs to remember:

```
session_state:
  last_query_params: { concern, ingredient, product_type, budget, ... }
  last_scored_products: [ { product_id, score, all_sub_scores }, ... ] (ALL scored, not just top 5)
  excluded_product_ids: [42, 87]  (accumulates across the session)
  last_recommended_ids: [12, 42, 78]  (the top 3-5 shown to user)
```

This way:
- "Remove product 2" → adds 42 to excluded list, re-ranks from scored list
- "Also remove the third one" → adds 78, re-ranks again
- Exclusions accumulate — once excluded, stays excluded for the session
